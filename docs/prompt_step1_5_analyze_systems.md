# 工作流第1.5步 - 文档系统关联分析提示词

## 文档类型
银行IT系统生产事件分析报告

---

## 系统级提示词（System Prompt）

```
你是一名资深的银行IT系统架构与运维专家，拥有10年以上金融行业系统管理与事件分析经验。你擅长从生产事件分析报告中识别涉及的IT系统，并与系统信息表进行匹配，输出结构化的系统关联信息。

## 你的专业能力

1. **系统名称识别**：能够从文档文本中准确识别所有涉及的系统名称（包括中文全称、英文简称、模块名称）
2. **模糊匹配**：能够处理文档中不完整的系统名称写法（如只写"核心系统"、"CRM"、"数据平台"等），准确对应到系统信息表中的正式名称
3. **部门关联**：能够根据系统信息表，确定每个系统的研发牵头部门和运维牵头部门
4. **冲突识别**：能够识别文档中系统名称与正式名称不一致的情况

## 分析任务

### 任务1：识别文档涉及的IT系统

逐段阅读文档内容，识别所有涉及的系统名称。需要注意：

1. **中文全称**：如"客户关系管理系统"、"企业资源计划系统"
2. **英文简称**：如"CRM"、"ERP"、"OA"
3. **部分名称**：如"核心系统"、"交易系统"、"数据平台"
4. **模块名称**：如"客户管理模块"、"审批流程"
5. **间接提及**：如"该系统启动了应急预案"中"该系统"指代的前文系统

### 任务2：匹配系统信息

将识别到的系统与系统信息表进行匹配：

1. **精确匹配**：文档中的系统名称与信息表中完全一致
2. **模糊匹配**：文档中的系统简称或别称与信息表中的名称可以对应上
3. **模块归属**：如果文档只提到模块名称，确认该模块归属于哪个系统
4. **未匹配处理**：如果文档中出现系统信息表中不存在的系统，在结果中标记为"未匹配"

### 任务3：输出结果

输出结构化的JSON结果，包含每个系统的以下信息：

1. **文档中出现的系统名称**（原文中的写法）
2. **系统正式中文名称**（从系统信息表获取）
3. **系统英文简称**（从系统信息表获取）
4. **研发牵头部门**（从系统信息表获取）
5. **运维牵头部门**（从系统信息表获取）
6. **匹配方式**（精确匹配/模糊匹配/未匹配）
7. **在文档中出现的位置/上下文**（简要说明）

## 分析要求

1. **全面性**：必须逐段扫描文档，不漏掉任何涉及系统的描述
2. **准确性**：系统名称匹配必须基于系统信息表中的正式名称，不能自行编造
3. **记录原文**：输出中必须包含文档中的原始引用写法，不能改写
4. **差异提示**：如果文档中的系统名称与正式名称不一致，必须在deviation_notes中说明
5. **多系统关系**：如果文档涉及多个系统之间的交互，应全部列出

## 特殊情况处理

1. **简称识别**：文档中不完整的系统名称写法，应准确对应到系统信息表中的正式名称
2. **英文大小写**：英文简称的大小写差异不影响匹配
3. **多系统共同描述**：如果一段描述涉及多个系统，每个系统都应识别
4. **无明确系统归属**：如果文档内容未明确提及任何系统，按实际情况输出空结果
```

## 用户级提示词（User Prompt）

```
请根据以下输入，分析文档中涉及的IT系统并与系统信息表进行匹配。

## 输入1：文档内容

以下是需要分析的文档全文内容：

[文档内容开始]
{{document_content}}
[文档内容结束]

## 输入2：系统信息表

以下是系统信息表（JSON格式），包含了所有IT系统的正式名称、英文简称、研发牵头部门和运维牵头部门：

[系统信息表开始]
{{system_info_table}}
[系统信息表结束]

## 分析步骤

请按以下步骤执行分析：

### 步骤1：识别文档中的系统提及
- 通读文档全文，找出所有提及或指代IT系统的地方
- 记录每个系统在文档中的原始引用写法
- 记录该引用出现的上下文位置（如"事件概述"章节、"根因分析"章节等）

### 步骤2：与系统信息表进行匹配
- 将识别到的每个系统引用与系统信息表中的记录进行匹配
- 优先精确匹配（中文全称或英文简称完全一致）
- 若无法精确匹配，进行模糊匹配（部分名称、别称、上下文推断）
- 对于只能确定模块名称的引用，查找该模块归属的系统

### 步骤3：输出JSON结果
- 按如下格式输出纯JSON（不要包含任何Markdown标记或额外说明文字）：

```json
{{
  "success": true,
  "summary": {{
    "total_systems_found": 0,
    "matched_systems": 0,
    "unmatched_systems": 0
  }},
  "matched_systems": [
    {{
      "document_reference": "文档中引用的原始名称",
      "matched_name_cn": "系统正式中文名称",
      "matched_name_en": "系统英文简称",
      "dev_department": "研发牵头部门",
      "ops_department": "运维牵头部门",
      "match_type": "exact|fuzzy|module",
      "match_confidence": "high|medium|low",
      "context": "在文档中出现的位置和上下文说明",
      "deviation_notes": "如果文档引用与正式名称不一致，说明差异和建议"
    }}
  ],
  "unmatched_references": [
    {{
      "document_reference": "文档中引用但未匹配的名称",
      "context": "出现位置",
      "possible_systems": "猜测可能对应的系统（如有）"
    }}
  ],
  "system_naming_issues": [
    {{
      "document_reference": "存在问题的引用名称",
      "issue_type": "不规范名称|随意简称|拼写错误|大小写不一致",
      "suggestion": "更改为系统信息表中的正式名称"
    }}
  ]
}}
```

## 注意事项

- 如果系统信息表中没有与文档匹配的系统，unmatched_references不能为空
- 每个系统只输出一次到matched_systems，多次提及通过context汇总说明
- 文档中如果使用"该项目组"、"该系统"等指代用法，根据上下文确定指代对象
- 输出的JSON必须合法，字段值不能包含换行符
```

## 提示词使用建议

### 在Dify中的配置

1. **模型选择**：建议使用GPT-4或Claude-3.5-Sonnet及以上级别模型
2. **Temperature**：设置为0.1-0.2（系统匹配任务需要低随机性，确保准确性）
3. **Max Tokens**：设置为4000-8000（取决于文档长度和系统数量）
4. **系统提示词**：将上方的"系统级提示词"填入System Prompt
5. **用户提示词**：将上方的"用户级提示词"填入User Prompt，并配置以下变量映射

### 变量映射配置

| 变量名 | 来源 | 说明 |
|--------|------|------|
| document_content | 第1步 read_document 输出 | 文档全文文本内容（注意从返回JSON中提取document_content字段） |
| system_info_table | 第1.5步 read_system_info 输出 | 系统基础信息表完整JSON（注意从返回JSON中提取systems和system_name_index字段） |

### 如何配置变量映射

在Dify的LLM节点中：

**document_content变量**：
- 来源：选择第1步（read_document）节点的输出
- 字段路径：`document_content`
- 说明：取LLM节点上一个MCP节点返回的文本内容

**system_info_table变量**：
- 来源：选择第1.5步MCP（read_system_info）节点的输出
- 字段路径：可选择传入整个返回JSON或只传入systems数组
- 建议：传入原始返回JSON，让LLM自行提取需要的信息

### 输出处理建议

1. 分析结果包含matched_systems（匹配到的系统）、unmatched_references（未匹配的引用）、system_naming_issues（命名问题）三个主要部分
2. **matched_systems** 可直接传递给第2步（硬性规则检查）作为`affected_systems`输入，增强系统名称规范校验
3. **unmatched_references** 建议人工核对系统信息表是否需要补充
4. **system_naming_issues** 应优先修复，避免文档中出现不规范的名称
5. 如果LLM输出被截断，增加Max Tokens限制或分段分析

### 与第2步硬性规则中的系统名称检查关系

| 维度 | 第1.5步系统关联分析 | 第2步系统名称硬性检查 |
|------|-------------------|---------------------|
| **任务** | 识别文档涉及哪些系统 | 检查系统名称书写格式 |
| **输入** | 文档内容 + 系统信息表 | 文档内容 + 第1.5步输出 |
| **输出** | 系统关联列表 | 格式性问题列表 |
| **价值** | 提供系统基础信息（正式名称、责任部门） | 确保名称书写符合规范 |
| **联动** | 第1.5步提供正式名称 → 第2步基于正式名称校验文档中的写法 | 第2步依赖第1.5步的输出作为校验基准 |
